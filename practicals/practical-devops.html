Add CI with Github Actions

Before you make any changes, create a feature branch named ‚Äúeditable-number‚Äù to isolate your modifications from the main branch:

üíª git checkout -b editable-number

This should be the new default action that you take when working on your code to get you in the habit. When you start a feature, create a corresponding feature branch. Work until it is stable and only then merge it into the main branch.

Create the CI config file

GitHub actions allow us to execute a wide variety of actions in response to git events. If you are looking at the repository on Github, you will see there is an Actions tab, which will allow you to add a number of different actions designed by GitHub and the community. However, all these do is create a special configuration file in your repository, so we will do that directly.

At the top level of your package directory, you should have a .github directory with a workflows directory inside (if not, you should create them). Inside of workflows, create a new file called node.js.yml (the important part of the file name is the .yml on the end, but use the full name) with the contents shown below.

YAML (pronounced like ‚Äúcamel‚Äù) is a popular file format for text-based configuration files. It won‚Äôt be important to understand all of its ins and outs, for now the key thing to know is that like Python, indentation matters.

name: Node.js CI

on:
  push:
    branches: [ "main" ]
  pull_request:
    branches: [ "main" ]

jobs:
  build:

    runs-on: ubuntu-latest

    steps:
    - uses: actions/checkout@v3
    - name: Use Node.js 18.x
      uses: actions/setup-node@v3
      with:
        node-version: 18.x
        cache: 'npm'
    - run: npm ci
    - run: npm test

There are two main sections here: on and jobs. The on section tells GitHub Actions when to execute this action (when we push to main or make a PR to main). The jobs section describes the steps to carry out. In this case, it

  1. Checks out our code into a linux virtual machine
  2. Installs node 18
  3. Runs npm ci, which uses the package-lock.json file to install all of the dependencies
  4. Runs npm test

Add a status badge to your README.md

As you browse different repositories on GitHub, you may have seen flashy status badges in the README files that indicate that the tests are passing (or not). You can add one to your project as well.

As described in the documentation, the URL of the image is:

https://github.com/<OWNER>/<REPOSITORY>/actions/workflows/<WORKFLOW_FILE>/badge.svg

Create a PR

Let‚Äôs commit this change and push it to GitHub:

üíª git add .
üíª git commit -m "Add actions config file"
üíª git push -u origin editable-number

The last line tells the remote repository (origin) that we would like to create a new branch called editable-number, and connects the two for us.

Visit the repository on GitHub. There should be a yellow banner across the top notifying you of a change to a branch and asking if you would like to ‚ÄúCompare & pull request‚Äù. Do so. If you don‚Äôt see the banner, you can always create a new PR from the ‚ÄòPull requests‚Äô tab.

On the new page that comes up you will see a form for starting the PR. Type in a message and click the button to create the PR.

You will see a message that it is OK to merge with main. This is unsurprising since no one has done anything to main since you made a branch, so the merge just involved adding the changes in this branch on top of main. If you wait a moment, however, you will see a new message appear as GitHub Actions kicks off (there will be two: one from the classroom auto-grader and the one you just added). Give it a moment to run, and then it should come back and tell you that the checks have failed. This is also unsurprising since you haven‚Äôt changed the code yet. If you click on Details you can see the run log and the output from running the tests.

Enhancing the color picker with controllable number

Now let‚Äôs make our change the the ColorPicker.

Start the development server (üíª npm run dev) so you can see the effects of your changes. You should see the familiar ColorPicker appear in the web view. You can also run the tests with üíª npm test, and they should fail.

You are now ready to make the changes to the LabeledSlider. The change is very simple, you are going to remove the current span which reports the value (<span>{value}</span>) and replace it with a number input element. The numerical input will be very similar to its slider counterpart. Its value should be controlled by the React state (so that all the elements that depend on or modify that value remain in sync), it should update the color component as specified, and have appropriate min and max values. The difference is it will be of type ‚Äúnumber‚Äù.

You should see your new inputs at the end of the sliders, and the tests should all pass.

Revisit the PR

Now that you have working code, add, commit and push it with git.

Return to the PR page on GitHub and after a moment you should see the action kick off again. This time you should get a message that ‚ÄúAll checks have passed‚Äù. If not, make sure that the tests were passing locally and that the new code has actually been pushed to the branch.

Click the ‚ÄúMerge pull request‚Äù button. it should give you a chance to add a message to the merge and then you can click ‚ÄúConfirm merge‚Äù.

After the merge is complete, it will give you a ‚ÄúDelete branch‚Äù button. Click it to remove the feature branch from GitHub.

Clean up your local repository after the merge

Return to your terminal and switch back to the ‚Äúmain‚Äù branch (üíª git checkout main). You are going to want to pull the changes from GitHub, pruning to delete remote tracking references and then delete your local branch (üíª git pull --prune).

To avoid mistakes when deleting branches, you can add the --dry-run option to the pull --prune command , e.g. üíª git pull --prune --dry-run to double check before actually pruning anything (note you will still need to run the actual command when you are sure it correct).

The full command sequence is:

üíª git checkout main
üíª git pull --prune
üíª git branch -d editable-number

Deploy to Fly

We are going to use the Fly.io to deploy our applications. You should have already received an invitation to the ‚Äúcsci312a-s23‚Äù organization (via your Middlebury e-mail). The shared organization will enable us to all work under an account the department pays for. Note, Fly.io e-mails seem to end up in quarantine. If you don‚Äôt see an invitation, check your junk mail folder and any quarantined messages.

  1. Start by downloading the flyctl tool for your system from here
  2. If you already have an account sign in, if not, sign up for a free account. You should have already been invited to the csci312a-s23 organization and so should not need to enter a credit card to launch applications under that organization.
  3. Once you have signed up or signed in, you can prepare to deploy your application with üíª flyctl launch. You should generally choose the default options for the subsequent questions, with a location near us, and the ‚Äúcsci312a-s23‚Äù organization but do not choose to deploy right now.
  4. The launch command will have created several files for you: Dockerfile (and corresponding .dockerignore) and fly.toml. We will need to edit the Dockerfile for our application. Specifically:
    1. One lines 2 change FROM node:16-alpine AS builder to FROM node:18-alpine AS builder and on line 25 change FROM node:16-alpine AS runner to FROM node:18-alpine AS runner. This ensures we are using the same Node version on Fly.io as we are using locally.
    2. In multiple locations you will find commands for the yarn package manager. Comment those out and uncomment the corresponding lines for using npm (included in the file). We are using npm not yarn.
  5. You are now ready to deploy your application by executing üíª flyctl deploy.
  6. When the deployment is complete execute üíª flyctl open to use your application!
